# BetterEmail V2 - Project Execution Plan

## Project Goal
Transform the current prototype into a production-grade, AI-powered Gmail extension. The system will feature smart follow-ups, semantic search, priority sorting, and lead generation, utilizing a robust tech stack (Express, React, Supabase, MongoDB, OpenAI/Gemini/Claude).

## Core Principles
1.  **Outcome-Based**: Every phase delivers a fully working, tested feature.
2.  **Test-First/Test-Driven**: Features are verified via Unit Tests (Jest) and Localhost manual verification before moving forward.
3.  **Parallel Execution**: The plan is split into "Backend" and "Extension/Frontend" tracks where possible to allow two developers to work simultaneously.
4.  **Production Quality**: strictly typed (JSDoc/TS), linted, and modular.

## Phase 1: Smart Follow-up Logic (True "No Reply" Detection)
**Current State**: Manual timer-based reminders.
**Goal**: Automate reminders by checking if a reply actually exists.

### Backend Track (Person A)
- [ ] **Gmail API Integration**: Implement Oauth2 flow to read user's sent messages and thread history.
- [ ] **Logic Endpoint**: `GET /api/reminders/status/:threadId`. Checks if the last message in thread is from user (no reply).
- [ ] **Unit Test**: Mock Gmail API response. Test logic:
  - If last msg from me -> Reminder needed.
  - If last msg from other -> No reminder.

### Extension Track (Person B)
- [ ] **Thread Tracking**: Content script captures `threadId` from URL or DOM when sending.
- [ ] **Sync**: Periodically poll backend for status of tracked threads.
- [ ] **UI Update**: Show "Reply received" or "Still waiting" status in Popup.

**Verification**:
- *Localhost*: Send email to self. backend correctly identifies "no reply". Reply to self. Backend identifies "replied".

---

## Phase 2: Reply Reminders (Incoming Emails)
**Goal**: Nudge user to reply to important incoming emails.

### Backend Track (Person A)
- [ ] **Priority Algorithm**: Implement basic heuristic (e.g., question mark detection, sender frequency) to flag "Needs Reply".
- [ ] **Endpoint**: `GET /api/inbox/needs-reply`.
- [ ] **Unit Test**: Feed sample emails, assert correct flagging.

### Extension Track (Person B)
- [ ] **Inbox Injection**: Inject "Needs Reply" badge into Gmail Inbox list views.
- [ ] **Notification**: Browser notification if high-priority email sits for >24h.

**Verification**:
- *Localhost*: Receive email. Verify endpoint flags it. Verify badge appears in Gmail UI.

---

## Phase 3: Semantic Search (The "Killer Feature")
**Goal**: "Email from 2 months ago about school club" finds the exact email.

### Backend Track (Person A)
- [ ] **Ingestion Pipeline**: Background job to fetch past 30 days of emails.
- [ ] **Embedding**: Send email body to OpenAI `text-embedding-3-large`.
- [ ] **Vector Store**: Save vector + metadata to MongoDB.
- [ ] **Search Endpoint**: `POST /api/search` -> Embed query -> Vector Search -> Return ThreadIDs.
- [ ] **Unit Test**: Mock OpenAI. Verify vector storage format.

### Extension Track (Person B)
- [ ] **Search Bar**: Add global search bar to Extension Popup or inject into Gmail Top Bar.
- [ ] **Results UI**: Display search results with "Why this matched" context.
- [ ] **Link**: Clicking result opens specific Gmail thread.

**Verification**:
- *Localhost*: Index dummy emails. query "Banana". Ensure email containing "Fruit" (related) appears if semantic, or rigorous exact match if not.

### Backend Track (Person A)
